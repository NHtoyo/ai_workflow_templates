# エラー調査ワークフロー

## 基本方針

エラーが出たら、すぐに大改造しない。

まず観察、原因候補、確認方法、修正方針に分ける。

## Codexへ委任する内容

- エラーログの読み取り
- 関連ファイルの特定
- 原因候補の列挙
- 各候補の根拠整理
- 確認コマンドの提案
- 安全な確認コマンドの実行
- 修正方針の提示

## 初回調査の制約

- まずファイルは変更しない。
- プロジェクト全体を読まない。
- エラーログと関連しそうなファイルから見る。
- 必要になったら追加で読むファイルを提案する。

## Codex向けプロンプト

```text
エラー原因を調査してください。
まずファイルは変更しないでください。

目的：
Antigravity側で次に何を直すべきか判断できるようにすること。

手順：
1. エラーログを読む
2. 原因候補を最大3つ出す
3. 各候補の根拠を示す
4. 追加で確認すべきファイル・コマンドを示す
5. 安全な確認コマンドだけ実行する
6. 修正対象ファイルと修正方針を示す
7. 最後に、Antigravity側への指示文を作る

制約：
- まだファイルを変更しない
- プロジェクト全体を読まない
- 必要なファイルだけ読む
- 日本語で説明する
- 初心者向けに専門用語を短く補足する
```
