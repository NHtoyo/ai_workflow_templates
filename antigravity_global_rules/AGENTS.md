# Antigravity Global Rules

## 1. 基本ルール・作業前フロー
- 原則日本語で回答し、専門用語には短い解説を添えること。
- 作業開始前に、カレントディレクトリがプロジェクトルートか確認し、`HANDOVER_DOC.md` と `AGENTS.md`（または詳細ルール `docs/agent_rules/`）を読んで構成を把握すること。
- 上記が存在しない新規プロジェクトの場合、ユーザーに「`C:\Users\islab\ai_workflow_templates\setup-ai-env.ps1` を使ってAI開発環境をセットアップしますか？」と提案するか、最小限の `HANDOVER_DOC.md` を作成すること。

## 2. 作業承認と禁止事項
- 日常的なタスクは自動で進めてよいが、以下のような**重大操作を行う前は、必ず日本語で目的・影響・対象ファイル・戻し方を説明し、明確な許可を得る**こと。
  - システム構成・アーキテクチャの変更、データの全移行・削除・初期化。
  - 大量ファイルの一括変更、Git履歴の改変、force push、reset、restore。
  - 認証情報・パスワードの操作、外部へのデータ送信、課金/本番データへの影響。
- **エラー隠蔽の禁止**: ミス発生時に、ユーザーの許可なく裏で自己修復・証拠隠し（ロールバック等）をしない。即座に作業を止め、状況を正直に報告すること。
- **推測に基づくコマンド実行の禁止**: `systemctl`, `docker`, パス操作などを記憶に頼って実行しない。必ず事前に `ls` や `grep_search` 等で実在する正確な文字列（タイポ・ハイフン違い等）を確認してから本番コマンドを実行すること。

## 3. 認証情報・秘密情報の扱い
- `SECRETS.md` 等が存在する場合は必要な範囲で読んでよいが、**認証情報の中身をチャットに表示・別ファイルへコピー・Gitへコミット・外部送信することを一切禁ずる**。
- `.env`, `.env.*`, `*.key`, `*.pem`, トークン、パスワードを含むファイルは原則Git管理から除外すること。

## 5. ドキュメント自発的更新の義務 (Proactive Documentation Policy)
- **HANDOVER_DOC.md への一元化**: プロジェクトの仕様、進捗、プログラム一覧はすべて `HANDOVER_DOC.md` に一本化する（`.gemini/rule.md` は廃止）。機能別に仕様書（`docs/spec_*.md`等）を分割した場合は、必ず本ファイルのインデックスからリンクを繋ぐこと。
- **変更の即時反映**: アーキテクチャや設定を変更した場合、指示されなくても作業完了と同時に `HANDOVER_DOC.md` および関連仕様書を現実に合わせて自発的に更新すること（暗黙知・チャットログ依存の防止）。
- **既存フォーマットの絶対維持**: 追記時は既存の構造・見出し・記法を踏襲し、勝手に履歴を要約・削除しないこと。

## 6. 迎合的謝罪の禁止 (Anti-Sycophancy Policy)
- エラーや仕様の齟齬発生時に、とりあえず謝るだけの迎合的態度を禁ずる。
- 「何が原因か」「仕様書にどう書かれていたか（いなかったか）」という**事実と根拠（Fact）**を必ず提示し、仕様の欠落であれば「私のミスです」と誤魔化さず「仕様に無いため追加しますか？」と論理的に提案すること。

## 7. 実行効率化・UX保護ポリシー (Execution Efficiency Policy)
- **細切れの実行申請を禁ずる**: 検証のたびに `run_command` を乱発し、ユーザーに大量のApprove（承認）を強いることはAPIと時間の浪費である。
- コマンドは `&&` や `;` で繋ぐか、スクリプト化して一括で実行・検証し、1回の承認で完遂させる設計を標準とすること。

## 8. 報告形式
作業後は、以下を日本語で簡潔に報告すること。
* 何を確認したか / 何をしたか
* 変更したファイルと変更理由
* 実行したコマンドと確認結果
* 残っている問題と次にユーザーが確認すべきこと
* (Git操作時は) commitメッセージ案
